# Аналіз до Лабораторної роботи №4: Синхронна та асинхронна комунікація

## 1. Що змінилося порівняно з лабораторною 3?
У третій лабораторній роботі наша система мала лінійний потік виконання: `Controller -> Command Handler -> Domain -> Repository`. Уся логіка виконувалася в одному контексті, і основна операція вважалася завершеною лише тоді, коли завершувався весь код усередині Handler-а. 

У цій лабораторній роботі ми змінили архітектурний підхід. Ми виділили побічні інфраструктурні та комунікаційні процеси, які не стосуються ядра домену, і делегували їх виконання окремим допоміжним компонентам. Це дозволило розвантажити основні Use Cases (Command Handlers) і застосувати патерни як прямої (синхронної) інтеграції, так і подійно-орієнтованої (асинхронної) комунікації за допомогою In-process Event Bus.

## 2. Виділені побічні операції
Під час аналізу юзкейсів системи було виділено дві основні побічні операції, які мають відбуватися у відповідь на бізнес-дії:

1. **Нотифікація (`NotificationService`)**: Сповіщення зацікавлених сторін (наприклад, студентів) про створення нової лабораторної роботи (`LabWork`).
   * *Чому виділено:* Відправка повідомлень (особливо email/push) — це повільна операція, що залежить від зовнішніх провайдерів. Вона не повинна блокувати збереження лабораторної в БД.
2. **Аудит логування (`AuditService`)**: Запис у системний журнал факту створення нової дисципліни (`Discipline`).
   * *Чому виділено:* Аудит є наскрізним функціоналом (cross-cutting concern). Бізнес-логіка створення дисципліни не повинна «знати» про те, куди і як пишуться логи безпеки чи аналітики.

Обидва компоненти винесені в окремі модулі та мають чіткі контракти (інтерфейси), що зберігає принцип інверсії залежностей (DIP).

## 3. Порівняння синхронного та асинхронного підходів

| Характеристика | Синхронний підхід | Асинхронний підхід (Event Bus) |
| :--- | :--- | :--- |
| **Час відповіді API** | Більший. Час відповіді складається з `DB Save` + `Side Effect Execution`. Клієнт чекає завершення всіх операцій. | Мінімальний. Клієнт отримує відповідь одразу після `DB Save`. Побічні ефекти виконуються у фоні без очікування. |
| **Поведінка при збоях** | Якщо `NotificationService` впаде, це може призвести до 500 помилки на всьому API (якщо не використати `try/catch`). Ми застосували `try/catch`, щоб ігнорувати помилку та не відкочувати транзакцію збереження в БД. | Збій у підписнику (Subscriber) ніяк не впливає на основний Handler. Основна операція успішно завершується, досягається повна ізоляція збоїв. |
| **Зв'язаність (Coupling)** | Середня. Handler має залежати від інтерфейсу допоміжного сервісу (напр. `INotificationService`), тобто знає про наявність конкретної побічної дії. | Дуже низька. Handler знає лише про `EventBus` і просто публікує подію (напр. `LabWorkCreated`). Він не знає, хто і як на це відреагує. |
| **Складність реалізації** | Низька. Простий виклик методу, легко дебажити (stack trace послідовний). | Вища. Потребує налаштування шини подій (Event Bus), написання підписників, ускладнює відстеження виконання (дебагінг). |

## 4. Вибір підходу для Production-системи
Для production-середовища я б обрала **Асинхронний підхід (Event-Driven Architecture)** з використанням зовнішнього брокера повідомлень (наприклад, RabbitMQ, Redis або Kafka) замість in-process Event Bus. 

**Чому?**
1. **Масштабованість:** Воркери (workers), які розсилають імейли або обробляють аналітику, можна масштабувати незалежно від основного API сервера.
2. **Стійкість (Resilience):** Якщо сервіс розсилки листів впаде, брокер повідомлень збереже подію `LabWorkCreated` у черзі (Dead Letter Queue) і повторить спробу пізніше. У синхронному варіанті з `try/catch` повідомлення буде втрачено назавжди.
3. **Eventual Consistency:** База даних оновлюється швидко, а інші системи підтягнуть зміни згодом. Синхронний підхід я б залишила **лише** для критичних транзакцій, де побічний ефект є невід'ємною частиною бізнес-вимоги (наприклад, списання коштів + запис у фінансовий леджер).

## 5. Подвійна доставка подій та Ідемпотентність
**Що станеться, якщо одна й та сама подія буде доставлена двічі?**
У системах обміну повідомленнями (брокерах) гарантується доставка "at-least-once" (хоча б один раз), тому дублювання подій у мережі — це часте явище. У нашій поточній реалізації, якщо подія `LabWorkCreated` надійде двічі, `NotificationService` просто **відправить два однакових сповіщення** студенту, а `AuditService` створить два однакових записи в логах. 

**Чи ідемпотентні наші обробники?**
Наразі — **ні**. Вони виконують дію за кожним викликом, не перевіряючи, чи ця дія вже була виконана.

**Як це виправити (шлях до ідемпотентності):**
Щоб зробити обробники ідемпотентними, необхідно додати до кожної події унікальний ідентифікатор (`eventId` або `Idempotency-Key`). 
При отриманні події підписник повинен перевірити в БД (або Redis), чи оброблявся вже цей `eventId`. 
* Якщо так — подія ігнорується (Return OK).
* Якщо ні — виконується відправка нотифікації, і `eventId` записується в БД як оброблений.
